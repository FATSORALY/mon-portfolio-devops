CI/CD avec GitLab CI : de zéro à la production

📅 5 Janvier 2026 • ⏱️ 12 min de lecture

GitLab CI est l'un des outils d'intégration continue les plus puissants et les plus simples à prendre en main. Dans cet article, nous allons construire une pipeline CI/CD complète pour une application Node.js, du commit jusqu'au déploiement sur un serveur de production.

Les bases de GitLab CI

GitLab CI utilise un fichier .gitlab-ci.yml à la racine de votre projet. Ce fichier définit :

  • Stages : les étapes de la pipeline (build, test, deploy)
  • Jobs : les tâches exécutées dans chaque stage
  • Runners : les agents qui exécutent les jobs

Structure d'une pipeline classique

stages:
  - build
  - test
  - package
  - deploy

variables:
  DOCKER_IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

before_script:
  - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY

Étape 1 : Build de l'application

build:
  stage: build
  image: node:18-alpine
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 hour

Étape 2 : Tests unitaires et linting

test:
  stage: test
  image: node:18-alpine
  script:
    - npm install
    - npm run lint
    - npm run test:unit
  coverage: '/All files[^|]*\|[^|]*\s+([\d\.]+)/'

Étape 3 : Construction de l'image Docker

package:
  stage: package
  image: docker:24
  services:
    - docker:dind
  script:
    - docker build -t $DOCKER_IMAGE .
    - docker push $DOCKER_IMAGE
  only:
    - main
    - develop

Étape 4 : Déploiement sur plusieurs environnements

deploy_staging:
  stage: deploy
  image: alpine:latest
  script:
    - apk add --no-cache openssh-client
    - ssh -o StrictHostKeyChecking=no $STAGING_USER@$STAGING_HOST "docker pull $DOCKER_IMAGE && docker-compose up -d"
  environment:
    name: staging
    url: https://staging.monapp.com
  only:
    - develop

deploy_production:
  stage: deploy
  image: alpine:latest
  script:
    - apk add --no-cache openssh-client
    - ssh -o StrictHostKeyChecking=no $PROD_USER@$PROD_HOST "docker pull $DOCKER_IMAGE && docker-compose up -d"
  environment:
    name: production
    url: https://monapp.com
  only:
    - main
  when: manual  # Approbation manuelle pour la prod

Pipeline complète avec tests d'intégration

stages:
  - build
  - test
  - security
  - package
  - integration
  - deploy

variables:
  DOCKER_IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
  SONAR_TOKEN: $SONAR_TOKEN

cache:
  paths:
    - node_modules/

build:
  stage: build
  image: node:18-alpine
  script:
    - npm ci --cache .npm --prefer-offline
    - npm run build
  cache:
    paths:
      - node_modules/
  artifacts:
    paths:
      - dist/

test:
  stage: test
  image: node:18-alpine
  script:
    - npm run test:unit
    - npm run test:integration

security:
  stage: security
  image: sonarsource/sonar-scanner-cli
  script:
    - sonar-scanner -Dsonar.projectKey=$CI_PROJECT_NAME -Dsonar.sources=. -Dsonar.host.url=$SONAR_HOST_URL -Dsonar.login=$SONAR_TOKEN

package:
  stage: package
  image: docker:24
  services:
    - docker:dind
  script:
    - docker build --build-arg VERSION=$CI_COMMIT_SHORT_SHA -t $DOCKER_IMAGE .
    - docker push $DOCKER_IMAGE

integration:
  stage: integration
  image: docker:24
  services:
    - docker:dind
  script:
    - docker run -d -p 3000:3000 --name test-app $DOCKER_IMAGE
    - sleep 10
    - curl --fail http://localhost:3000/health
  after_script:
    - docker rm -f test-app

deploy_staging:
  stage: deploy
  image: alpine:latest
  script:
    - apk add --no-cache openssh-client
    - ssh $STAGING_USER@$STAGING_HOST "docker pull $DOCKER_IMAGE && docker-compose up -d"
  environment: staging
  only:
    - develop

deploy_production:
  stage: deploy
  image: alpine:latest
  script:
    - apk add --no-cache openssh-client
    - ssh $PROD_USER@$PROD_HOST "docker pull $DOCKER_IMAGE && docker-compose up -d"
  environment: production
  only:
    - main
  when: manual

Utilisation des variables d'environnement

Dans GitLab UI : Settings → CI/CD → Variables

# Variables à configurer :
STAGING_HOST    # IP du serveur de staging
STAGING_USER    # utilisateur SSH
PROD_HOST       # IP du serveur de production  
PROD_USER       # utilisateur SSH
SONAR_TOKEN     # Token SonarQube
CI_REGISTRY     # URL du registry Docker
CI_REGISTRY_USER
CI_REGISTRY_PASSWORD

Dockerfile pour l'application

# Build stage
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

# Production stage
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package*.json ./
EXPOSE 3000
CMD ["node", "dist/index.js"]

Bonnes pratiques GitLab CI

  • ✅ Utiliser des caches pour accélérer les pipelines
  • ✅ Séparer les stages logiquement (build → test → deploy)
  • ✅ Utiliser needs pour exécuter des jobs en parallèle
  • ✅ Activer les tests de sécurité (SAST, DAST)
  • ✅ Stocker les secrets dans les variables protégées
  • ✅ Utiliser rules au lieu de only/except (plus flexibles)

Exemple avec needs (parallélisation)

stages:
  - build
  - test
  - deploy

build:
  stage: build

test:unit:
  stage: test
  needs: ["build"]

test:integration:
  stage: test
  needs: ["build"]

deploy:
  stage: deploy
  needs: ["test:unit", "test:integration"]

Vous avez maintenant toutes les bases pour mettre en place une pipeline CI/CD professionnelle avec GitLab CI. Commencez simple, puis ajoutez progressivement des étapes de sécurité, de monitoring et de tests avancés !
