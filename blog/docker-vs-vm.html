Docker vs VM : ce que j'ai appris en pratique

📅 15 Décembre 2025 • ⏱️ 9 min de lecture

Quand j'ai commencé dans le DevOps, je me suis posé la question : Docker ou machine virtuelle ? Aujourd'hui, après plusieurs projets en production, je partage mon expérience sur les différences concrètes, les cas d'usage et comment choisir.

La différence fondamentale

La principale différence réside dans l'architecture :

  • VM : Virtualisation complète du matériel → chaque VM a son propre OS
  • Docker : Virtualisation au niveau OS → partage le noyau de l'hôte

Tableau comparatif détaillé

Ce que j'ai mesuré en pratique

Tests réalisés sur mon poste de développement (16GB RAM, 8 cores) :

  • Démarrage : VM Ubuntu → 45s / Conteneur Nginx → 0.8s
  • Consommation mémoire : VM (1GB alloués) / Conteneur (18MB)
  • Stockage : VM 4GB / Image Docker 80MB
  • Déploiement : 10 VM en 5 min / 50 conteneurs en 30s

Avantages concrets de Docker

1. Rapidité de développement

# Avec Docker
docker run -p 3000:3000 -v $(pwd):/app node:18 npm run dev

# Environnement identique en dev, staging, prod

2. Reproductibilité parfaite

# Dockerfile qui garantit l'environnement exact
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]

3. CI/CD simplifié

# Une seule commande pour déployer
docker build -t monapp .
docker push monapp
# Sur le serveur
docker pull monapp && docker run monapp

Quand utiliser des VMs ?

  • ✅ Isolation stricte : sécurité maximale, clients multi-tenant
  • ✅ OS différents : besoin d'exécuter Windows ET Linux sur le même hyperviseur
  • ✅ Environnements legacy : anciennes applications avec dépendances système complexes
  • ✅ Kernel spécifique : besoin de modules noyau personnalisés

Quand privilégier Docker ?

  • ✅ Microservices : décomposer l'application en petits services
  • ✅ CI/CD pipelines : exécution de jobs isolés et rapides
  • ✅ Dev vs Prod : environnement identique partout
  • ✅ Scalabilité horizontale : Kubernetes, Docker Swarm
  • ✅ Resource efficiency : optimiser le coût cloud

Cas hybride : Docker dans une VM (meilleur des deux mondes)

En production cloud, la pratique courante :

# Infrastructure typique
Bare Metal / Cloud
  └── VM (avec Docker installé)
        ├── Conteneur App 1
        ├── Conteneur App 2
        └── Conteneur Base de données

Avantages : isolation de la VM + légèreté et orchestration de Docker

Exemple pratique : Déployer une app avec Docker

# docker-compose.yml pour une stack complète
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    depends_on:
      - postgres
      - redis

  postgres:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data

  redis:
    image: redis:7-alpine

volumes:
  pgdata:

Ma conclusion après 1 an d'expérience

  • Pour le développement : Docker obligatoire (gain de temps énorme)
  • Pour la production : Docker + orchestration (K8s) pour la majorité des cas
  • Pour la sécurité stricte : VM ou combinaison VM + Docker

Règle d'or : Commencez avec Docker, passez aux VMs uniquement si vous avez des contraintes spécifiques (sécurité, noyau, OS).

Les pièges à éviter avec Docker

  • ⚠️ Ne pas stocker de données dans les conteneurs (utiliser des volumes)
  • ⚠️ Éviter les images trop grosses (préférer Alpine, slim)
  • ⚠️ Ne pas exécuter les conteneurs en root
  • ⚠️ Gérer les logs (ne pas laisser les conteneurs écrire trop de logs)
