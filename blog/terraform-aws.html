Terraform en pratique : provisionner AWS en 30 minutes

📅 10 Février 2026 • ⏱️ 10 min de lecture

Terraform est l'outil d'Infrastructure as Code (IaC) le plus populaire. Dans cet article, je vais vous montrer comment provisionner une infrastructure AWS complète en moins de 30 minutes.

Pourquoi Terraform ?

Terraform permet de décrire votre infrastructure dans des fichiers de configuration déclaratifs. Avantages :

  • ✅ Versionner votre infrastructure (Git)
  • ✅ Reproduire facilement des environnements (dev/staging/prod)
  • ✅ Éviter les erreurs manuelles
  • ✅ Gérer l'état de votre infrastructure

Prérequis

  • Un compte AWS (free tier suffit pour ce tuto)
  • Des clés d'accès AWS (Access Key ID + Secret Access Key)
  • Terraform installé sur votre machine

Installation de Terraform

# Linux / macOS
wget -O- https://apt.releases.hashicorp.com/gpg | gpg --dearmor | sudo tee /usr/share/keyrings/hashicorp-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] https://apt.releases.hashicorp.com $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/hashicorp.list
sudo apt update && sudo apt install terraform

# Vérifier l'installation
terraform version

Notre infrastructure cible

Nous allons créer :

  • 1 VPC avec un CIDR block 10.0.0.0/16
  • 2 sous-réseaux publics (1 par AZ)
  • 1 Internet Gateway
  • 1 Security Group (SSH + HTTP)
  • 2 instances EC2 t2.micro
  • 1 Load Balancer

Structure du projet

terraform-aws/
├── main.tf          # Ressources principales
├── variables.tf     # Variables
├── outputs.tf       # Sorties
└── terraform.tfvars # Valeurs des variables (ignoré par Git)

Le fichier main.tf

# Provider AWS
provider "aws" {
  region = var.aws_region
}

# VPC
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name = "terraform-vpc"
  }
}

# Internet Gateway
resource "aws_internet_gateway" "gw" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "terraform-igw"
  }
}

# Sous-réseaux publics
resource "aws_subnet" "public" {
  count                   = 2
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.${count.index}.0/24"
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name = "public-subnet-${count.index + 1}"
  }
}

# Security Group
resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Allow SSH and HTTP"
  vpc_id      = aws_vpc.main.id

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "SSH"
  }

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTP"
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# Instances EC2
resource "aws_instance" "web" {
  count         = 2
  ami           = data.aws_ami.amazon_linux_2.id
  instance_type = "t2.micro"
  subnet_id     = aws_subnet.public[count.index].id
  vpc_security_group_ids = [aws_security_group.web.id]

  user_data = <<-EOF
    #!/bin/bash
    yum update -y
    yum install -y httpd
    systemctl start httpd
    systemctl enable httpd
    echo "<h1>Hello from Terraform - Instance ${count.index + 1}</h1>" > /var/www/html/index.html
  EOF

  tags = {
    Name = "web-server-${count.index + 1}"
  }
}

# Données utiles
data "aws_availability_zones" "available" {
  state = "available"
}

data "aws_ami" "amazon_linux_2" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

Le fichier variables.tf

variable "aws_region" {
  description = "Région AWS"
  type        = string
  default     = "eu-west-3"  # Paris
}

Déploiement de l'infrastructure

# Initialiser Terraform (télécharge les providers)
terraform init

# Voir ce qui va être créé
terraform plan

# Appliquer la configuration
terraform apply

# Détruire l'infrastructure (quand vous avez fini)
terraform destroy

Bonnes pratiques à adopter

  • State remote : stocker le fichier terraform.tfstate sur S3 + DynamoDB pour le locking
  • Modules : réutiliser des modules pour ne pas se répéter
  • Workspaces : gérer plusieurs environnements (dev, staging, prod)
  • terraform fmt : formater le code automatiquement
  • terraform validate : vérifier la syntaxe

Configuration du remote state (S3 + DynamoDB)

# backend.tf
terraform {
  backend "s3" {
    bucket         = "mon-bucket-terraform-state"
    key            = "dev/terraform.tfstate"
    region         = "eu-west-3"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}

Avec ces bases, vous pouvez maintenant provisionner une infrastructure AWS complète en quelques minutes, de façon reproductible et fiable. La prochaine étape sera d'ajouter un cluster Kubernetes managé (EKS) avec Terraform !
